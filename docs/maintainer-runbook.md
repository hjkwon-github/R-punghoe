# 메인테이너 운영 가이드 (Runbook)

이 문서는 R-punghoe(알풍회) 위키를 운영하는 메인테이너가 일상적으로 따르는 절차를 단계별로
정리합니다. 설계 배경은 [architecture.md](./architecture.md), 기록 형식은 [content-model.md](./content-model.md),
배포 세부는 [deployment.md](./deployment.md), 정정·삭제 정책은 [policy.md](./policy.md)를 함께 보세요.

운영의 대전제: **공개 출처만, 메타데이터 4종(날짜·출처·타임스탬프·검증 상태)과 함께, 차분하고
사실 위주로.**

## 1. 새 기록 추가하기

기록 1건은 파일 1개로 관리합니다. 절차는 다음과 같습니다.

1. `main`에서 새 브랜치를 만듭니다. (외부 기여자는 fork 후 PR)
   ```bash
   git switch -c record/<짧은-주제명>
   ```
2. `src/content/docs/archive/records/` 아래에 새 Markdown 파일을 만듭니다. 파일명은 영어 소문자와
   하이픈을 권장합니다. (예: `2024-09-altsangmu-interview.md`)
3. 프런트매터에 필수 메타데이터를 채웁니다. 정확한 필드 정의는 [content-model.md](./content-model.md)를
   따르되, 최소한 다음을 포함합니다.
   ```md
   ---
   title: 기록 제목
   date: 2024-09-01            # 사건/발언이 있었던 날짜
   source: https://...         # 공개 출처 URL
   timestamp: "12:34"          # 영상이라면 해당 시점 (없으면 비움 가능)
   verification: 검토중         # 확인됨 / 검토중 / 보류 중 하나
   tags: [영상, 발언]
   ---
   ```
4. 본문에는 사실 요약과 맥락을 적습니다. 영상 전체 전사·원문 통째 복사는 금지이며, 출처 링크로
   대체합니다.
5. 필요하면 연대기(`archive/timeline.md`)와 용어사전(`archive/glossary.md`)에 교차 링크를 추가합니다.
6. 로컬에서 빌드가 깨지지 않는지 확인합니다.
   ```bash
   npm run build
   ```
7. 커밋하고 PR을 엽니다. PR 본문에 출처 링크와 검증 상태 근거를 적습니다.

> 새 기록은 보통 `검토중`으로 시작합니다. 1차 공개 출처가 확실히 확인된 경우에만 `확인됨`으로
> 올립니다.

## 2. PR 리뷰 체크리스트

PR을 병합하기 전에 아래 항목을 모두 확인합니다. 하나라도 걸리면 병합하지 않고 코멘트로 수정을
요청합니다.

- [ ] **출처가 공개 자료인가?** 유튜브·기사·공식 채널·공개 인터뷰 등 누구나 접근 가능한 출처인가.
- [ ] **메타데이터 4종이 있는가?** 날짜·출처(URL)·타임스탬프(해당 시)·검증 상태가 채워졌는가.
- [ ] **검증 상태 값이 정확한가?** `확인됨`/`검토중`/`보류` 셋 중 하나인가 (오타·다른 값 금지).
- [ ] **출처 링크가 실제로 열리는가?** URL이 살아 있고 본문 주장과 일치하는가.
- [ ] **금지 항목이 없는가?**
  - 비공개 개인정보(가족·주소·연락처·사적 동선) 없음.
  - 출처 없는 루머·추측 없음.
  - 특정 개인 비방·조롱·공격성 표현 없음.
  - 투자 매수/매도/보유 권유로 읽힐 표현 없음.
  - 영상 전체 전사·과도한 저작물 복제 없음.
- [ ] **톤이 차분하고 사실 위주인가?** 밈을 보존하더라도 개인을 깎아내리지 않는가.
- [ ] **빌드가 통과하는가?** `npm run build`가 에러 없이 끝나는가 (프런트매터 스키마 포함).
- [ ] **링크/교차참조가 올바른가?** 연대기·용어사전 연결이 깨지지 않는가.

체크리스트를 통과하면 병합합니다. 병합되면 Cloudflare Pages가 자동으로 빌드·배포합니다
(→ [deployment.md](./deployment.md)).

## 3. 검증 상태 전환 기준

검증 상태는 정확히 세 가지이며, 전환 기준은 다음과 같습니다.

| 상태 | 의미 | 들어가는 조건 |
| --- | --- | --- |
| `검토중` | 출처는 있으나 맥락 확인이 더 필요 | 새 기록 기본값. 출처는 달렸지만 내용·맥락을 아직 못 박은 상태 |
| `확인됨` | 1차 공개 출처로 확인된 기록 | 살아 있는 공개 출처에서 본문 주장이 직접 확인됨. 타임스탬프(있는 경우) 일치 |
| `보류` | 근거 부족으로 본문 편입 전 | 출처가 불충분·불명확하거나, 검토 결과 신뢰도가 낮아 노출을 미루는 경우 |

전환 규칙:

- `검토중 → 확인됨`: 출처를 직접 열어 본문과 대조한 뒤에만 올립니다. 의심이 남으면 올리지 않습니다.
- `검토중/확인됨 → 보류`: 출처가 사라졌거나(죽은 링크) 정정 요청으로 신뢰도가 흔들리면 즉시 내립니다.
- `보류 → 검토중/확인됨`: 새 공개 출처가 확보되면 다시 올릴 수 있습니다.

> 애매하면 더 보수적인 상태(검토중 또는 보류)를 택합니다. 아카이브의 신뢰가 노출 속도보다 우선합니다.

## 4. 배포와 롤백

배포는 자동입니다. `main`에 병합되면 Cloudflare Pages가 빌드해 CDN으로 배포합니다. 메인테이너가
직접 챙길 부분은 다음과 같습니다.

- **빌드 실패 시:** Cloudflare Pages 대시보드의 빌드 로그를 확인하고, 원인을 고치는 PR을 올립니다.
  보통 프런트매터 스키마 오류나 깨진 링크가 원인입니다.
- **잘못된 배포 롤백:** 문제 커밋을 되돌리는 PR(revert)을 병합하면 다음 빌드에서 정상 상태로
  돌아옵니다. 긴급 시 Cloudflare Pages에서 직전의 정상 배포로 즉시 되돌릴 수 있습니다.

> 구체적인 빌드 설정값, `_headers`(CSP) 내용, 롤백 단계별 절차는 [deployment.md](./deployment.md)를
> 정본으로 삼습니다.

## 5. 정정 · 삭제 요청 처리

정정/삭제 요청은 GitHub 이슈(정정 요청 템플릿)로 받습니다. 처리 흐름은 다음과 같습니다.

1. 요청 접수: 어떤 기록의 무엇을, 어떤 근거로 바꾸려는지 확인합니다.
2. 사실 확인: 제시된 출처 또는 기존 출처를 다시 검토합니다.
3. 조치: 정정(본문 수정), 검증 상태 하향(`보류`), 또는 삭제를 결정합니다. 사생활·비공개 정보·
   명백한 오류는 신속히 처리합니다.
4. 투명성: 처리 결과와 사유를 이슈/PR에 남겨 이력을 공개합니다.

> 정정/삭제의 판단 기준, 우선순위, 비공개 정보 대응 등 구체 정책은 [policy.md](./policy.md)를
> 정본으로 삼습니다.

## 6. 주기적 점검

다음 점검을 정기적으로 수행합니다. (Phase 3에서 일부는 CI로 자동화 예정 → [roadmap.md](./roadmap.md))

- **죽은 링크 점검:** 출처 URL이 여전히 열리는지 확인합니다. 사라진 출처가 있으면 해당 기록을
  `보류`로 내리고, 대체 출처를 찾거나 처리 이력을 남깁니다.
- **오래된 `검토중` 점검:** 오래 방치된 `검토중` 기록을 모아 다시 검토합니다. 확인되면 `확인됨`,
  근거가 안 나오면 `보류`로 정리합니다.
- **검증 상태 일관성:** `확인됨`/`검토중`/`보류` 외의 값이 들어간 기록이 없는지 확인합니다.
- **금지 항목 재점검:** 시간이 지나며 추가된 본문에 사생활·루머·투자권유·비방 표현이 섞이지
  않았는지 표본 점검합니다.
- **빌드 건강성:** 정기적으로 `npm run build`가 통과하는지, 검색(Pagefind)·사이트맵이 정상
  생성되는지 확인합니다.
