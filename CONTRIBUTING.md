# 기여 가이드 (Contributing)

R-punghoe(알풍회)는 알상무의 **공개 활동·영상·발언·밈·자료 흔적**을 출처와 함께 정리하는 **비공식 팬 아카이브**입니다. 이 프로젝트는 알상무 본인, 슈카친구들, 머니코믹스와 공식 관계가 없습니다.

기여해 주셔서 감사합니다. 모든 기록은 **공개 출처**를 기준으로 하고, 차분하고 투명하며 사실에 충실한 태도를 지킵니다. 시작하기 전에 이 문서와 [행동 강령](./CODE_OF_CONDUCT.md), 그리고 사이트의 [기록 원칙](./src/content/docs/principles.md)을 함께 읽어 주세요.

> 이 저장소는 **정적 사이트(Astro + Starlight, Cloudflare Workers (정적 자산 / Static Assets))** 이며 데이터베이스나 서버가 없습니다. 모든 기여는 **GitHub Issue 또는 Pull Request**로만 받습니다.

---

## 1. 반드시 지켜야 하는 규칙 (Hard Rules)

다음에 해당하면 **반려**됩니다. 예외는 없습니다.

- ❌ **비공개 개인정보 금지** — 가족, 주소, 연락처, 직장 위치, 사적 동선 등 사생활 정보.
- ❌ **출처 없는 루머·추측 금지** — 확인되지 않은 소문, 카더라, 짜깁기.
- ❌ **공격·명예훼손 금지** — 특정 개인이나 집단을 비하·조롱·공격하는 내용.
- ❌ **투자 조언 금지** — 매수/매도/보유 권유, 종목·가격 추천 등 투자 권유로 읽히는 표현.
- ❌ **과도한 전재 금지** — 영상 전체 전사, 기사 본문 통째 복사 등 과도한 저작물 복제. 인용은 **짧게**, 맥락 요약 중심으로.

밈은 보존하되 **사람을 깎아내리거나 사생활을 추적하는 방식**으로는 다루지 않습니다.

---

## 2. 기여 흐름 한눈에 보기

```text
[제안할 기록이 있다]
        │
        ├─ 새 기록 제안        → Issue: "기록 제출" 템플릿
        ├─ 기존 기록 오류 정정 → Issue: "정정 요청" 템플릿
        └─ 삭제·수정(권리/사생활) → Issue: "삭제·정정 요청" 템플릿
        │
        ▼
[검토/논의]  →  [Pull Request 작성]  →  [검증 상태 확인]  →  [머지]
```

직접 글을 작성할 자신이 없어도 괜찮습니다. **Issue만 남겨 주셔도** 운영진/다른 기여자가 PR로 옮길 수 있습니다.

---

## 3. Issue로 기록 제안하기

저장소의 **[Issues](https://github.com/hjkwon-github/R-punghoe/issues/new/choose)** 에서 목적에 맞는 템플릿을 선택하세요.

- **기록 제출** (`record-submission`) — 새로운 공개 기록을 제안합니다. 제목, 날짜, 출처 URL, 타임스탬프, 검증 상태, 태그, 요약을 입력합니다.
- **정정 요청** (`correction-request`) — 기존 기록의 오류를 신고합니다. 대상 페이지, 무엇이 틀렸는지, 올바른 출처를 입력합니다.
- **삭제·정정 요청** (`takedown-request`) — 사생활·권리 문제로 내용 삭제/수정을 요청합니다. 대상 내용, 사유, 요청자 관계를 입력합니다. 정중하게 다루며 운영 정책에 따라 처리합니다.

Issue 템플릿은 [`.github/ISSUE_TEMPLATE/`](./.github/ISSUE_TEMPLATE)에 정의되어 있습니다.

## 4. Pull Request로 직접 기여하기

1. 저장소를 **포크(fork)** 합니다.
2. 작업 브랜치를 만듭니다. 예: `git checkout -b record/2024-shuka-live`
3. 아래 [기록 형식](#5-기록-형식-frontmatter)에 맞춰 문서를 작성/수정합니다.
4. 로컬에서 빌드가 깨지지 않는지 확인합니다.
   ```bash
   npm install
   npm run dev      # 미리보기 (http://localhost:4321)
   npm run build    # 정적 빌드 검증
   ```
5. 변경 사항을 커밋하고 푸시한 뒤 **Pull Request**를 엽니다.
6. PR 본문의 [체크리스트](./.github/PULL_REQUEST_TEMPLATE.md)를 빠짐없이 채우고, 관련 Issue를 링크합니다.

## 5. 기록 형식 (Frontmatter)

기록 1건은 `src/content/docs/archive/records/` 아래 **파일 1개**로 작성합니다. 프런트매터 필드 이름과 형식은 실제 스키마(`src/content.config.ts`)와 **정확히 일치**해야 합니다. 이름이 다르면 빌드는 통과해도 **메타데이터가 조용히 버려집니다.** 전체 설명은 [docs/content-model.md](./docs/content-model.md)를, 채워진 예시는 [example-record.md](./src/content/docs/archive/records/example-record.md)를 참고하세요.

```yaml
---
title: 항목 제목
description: 한 줄 설명 (검색·목록 표시용)
recordDate: 2024-05-17           # YYYY-MM-DD (확인된 공개 일자)
verification: 검토중              # 확인됨 / 검토중 / 보류 중 하나
recordTags:                      # 분류용 키워드 (문자열 배열)
  - 시장발언
  - 밈
sources:                         # 공개 출처 (객체 배열, 한 개 이상)
  - title: 영상/기사 제목 또는 채널명
    url: https://www.youtube.com/watch?v=XXXXXXXXXXX
    timestamp: "12:34"           # 영상 내 발언/장면 위치 (있으면)
related:                         # 관련 기록 slug (선택)
  - archive/records/other-record
---

본문에는 사실 요약과 맥락을 적습니다. 발언은 짧게만 인용합니다.
```

> ⚠️ `날짜/출처/타임스탬프/검증상태/태그` 같은 한글 키나 `date/source/tags` 같은 평면 키는 **스키마에 없습니다.** 반드시 위 필드명(`recordDate`, `verification`, `recordTags`, `sources[]`)을 사용하세요. `sources`는 평면 URL이 아니라 `title`/`url`/`timestamp`를 가진 **객체 배열**입니다.

### 검증 상태 값 (정확히 세 가지)

| 값 | 의미 |
| --- | --- |
| `확인됨` | 1차 공개 출처가 확인된 기록 |
| `검토중` | 출처는 있으나 맥락 확인이 더 필요한 기록 |
| `보류` | 출처가 불충분해 본문 편입 전인 기록 |

## 6. 검토·검증 흐름

1. **접수** — Issue 또는 PR로 제안이 들어옵니다.
2. **출처 확인** — 링크가 살아 있고, 공개 자료이며, 내용과 일치하는지 확인합니다.
3. **검증 상태 부여** — 위 기준에 따라 `확인됨 / 검토중 / 보류` 중 하나를 부여합니다.
4. **규칙 검토** — [반드시 지켜야 하는 규칙](#1-반드시-지켜야-하는-규칙-hard-rules)에 위배되는 부분이 없는지 확인합니다.
5. **머지 또는 보완 요청** — 문제가 없으면 머지하고, 보완이 필요하면 코멘트로 요청합니다.

`보류` 상태의 기록은 본문에 바로 노출되지 않을 수 있으며, 추가 출처가 확보되면 상태가 갱신됩니다.

## 7. 좋은 기여 / 반려될 수 있는 기여

**좋은 기여**

- 원본 링크와 정확한 타임스탬프가 있는 기록
- 발언을 길게 옮기지 않고 맥락을 요약한 기록
- 밈의 유래와 사용 맥락을 구분해 정리한 기록

**반려될 수 있는 기여**

- 출처 없는 소문, 사생활 침해 소지가 있는 내용
- 특정 개인·집단에 대한 공격이나 조롱
- 투자 매수/매도/보유를 권하는 문장
- 저작물의 과도한 복제(전체 전사 등)
